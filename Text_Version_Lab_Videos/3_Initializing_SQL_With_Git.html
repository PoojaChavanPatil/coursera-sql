Initializing a SQL Project with Git

Introduction

Hey data enthusiasts! Ever lost track of your SQL query versions or wished you could go back to a previous version of your code? In this video, we'll explore a game-changing tool for data analysts - Git version control for SQL projects. Just like how Google Docs tracks changes in documents, Git helps us manage our SQL code professionally. Let's learn how to never lose our query history again!

Learning Objectives

In this hands-on session, we'll:

  • Set up a Git repository for SQL projects
  • Learn how to track changes in our queries
  • Write professional commit messages
  • And implement version control best practices that real data teams use every day
  • Whether you're working solo or in a team, these skills are essential for modern data analysis.

Setting Up Git

Let's start by making sure Git is ready to use. First, we'll check our Git version by running the git version command. The output shows your installed version of git.

[Code begins here]

!git --version
[Code ends here]

Next, let's set up our Git configuration. If you're setting up Git for the first time, you'll need to tell Git who you are. This is crucial because in real teams, everyone needs to know who made which changes. Let's set that up:

Replace your name and email with the current text in the code block. These first two lines will set your name and email for Git. The third line will output a list of your current configuration. The output after running these lines should show your name and email address.

[Code begins here]

!git config --global user.name "Your Name"
!git config --global user.email "your.email@example.com"
!git config --list
[Code ends here]

Creating the Repository

Now, let's create a home for our SQL project. We'll call it 'bookcycle_queries'. To do this we'll run the make directory command followed by the name of the directory. Then we'll run the "cd" command with the new directory name. CD stands for Change directory and allows us to move into the new directory.

[Code begins here]

!mkdir bookcycle_queries
%cd bookcycle_queries
[Code ends here]

To see the directory you created, navigate to the side panel where you will find a folder named "bookcyle_queries". Notice the timestamp under "last Modified" label. This is the directory that you just created.

Now its time to initialize our Git repository by running the command "git init"

[Code begins here]

!git init
[Code ends here]

You should see now a message about initializing an empty repository. Think of this as creating a special folder that can track all changes to our SQL files.

Let's add a .gitignore file - this is like a VIP list, but in reverse. It tells Git which files NOT to track:

[Code begins here]

%%writefile .gitignore
*.db
*.sqlite
.ipynb_checkpoints/
[Code ends here]

In real projects, this prevents cluttering your repository with database files and temporary files.

Adding the First Query

Next Let's create our first SQL query file. This code is writing a file a called customeranslysis.sql Then inputting the information to include in the file, in this case a SQL query that finds customer ids and preferred story from members only, then orders the output by birth year.

[Code begins here]

%%writefile customer_analysis.sql
-- BookCycle Customer Analysis
SELECT customer_id,  preferred_store
FROM customer
WHERE is_member == 1
ORDER BY birth_year DESC;
SELECT customer_name, purchase_count
FROM customers
WHERE purchase_count > 5
ORDER BY purchase_count DESC;
[Code ends here]

To view the new file, navigate to the bookcyle_queries folder. This contains the customer_analysis.sql file, which contains the query entered in the above code cell. This analyzes our member most active customers - something you might actually do when identifying loyal customers for a marketing campaign.

Now, let's tell Git to add this file to its tracking by typing the git add command followed by the file name. We can output the status of this action by follow up with the git status command. The output gives us some important information about our tracking status:

[Code begins here]

!git add customer_analysis.sql

!git status
[Code ends here]

For example, the Green text means the file is staged - think of it as being in Git's waiting room, ready to be saved permanently.

Committing Changes

To save it permanently, we'll create a commit - A commit is like taking a snapshot of our code: To make a commit, we run the git commit -m command, followed by a message that briefly describes the action taken. In this case - Initial commit: Add customer analysis query.

[Code begins here]

!git commit -m "Initial commit: Add customer analysis query"
[Code ends here]

Now, let's update our original SQL query to pull the join date and birth year city information - a common real-world requirement when analyzing regional customer behavior. We'll run the write file command again, followed by our updated query that now pulls the customer ids, preferred stores and join dates for all customers born after 1990.

[Code begins here]

%%writefile customer_analysis.sql
-- BookCycle Customer Analysis
SELECT 
    customer_id,
    preferred_store,
    join_datecity
FROM customers
WHERE birth_year > 1990;

SELECT 
    customer_name,
    purchase_count,
    city
FROM customers
WHERE purchase_count > 5
ORDER BY purchase_count DESC;
[Code ends here]

You can check the customer_analysis.sql file again to see the code changes being reflected.

Let's track and commit these changes by running the git add and git commit -m commands again. This time our commit message reads "Add city information to customer analysis". This commit message clearly describes what changed. This is crucial when working in teams!

[Code begins here]

!git add customer_analysis.sql
!git commit -m "Add joining date information to customer analysis"
[Code ends here]

Lastly, let's run the git status command one more time. Notice how the output now shows that one file has been changed. This is the cool thing about git version control - every change is thoroughly tracked and documented.
